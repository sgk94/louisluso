Three Approaches for louisluso.com

Pick the architecture that fits your needs

A

Next.js + Zoho APIs (Recommended)

Custom-built Next.js site on Vercel. Server-side routes proxy all Zoho API calls (Inventory, Books, CRM). Clerk handles auth. Full control over every pixel.

Pros
  • Full design control — no theme limitations
  • Best performance (SSR + edge caching)
  • Your stack (Next.js, TypeScript, Vercel)
  • Zoho is just a data source — easy to swap later
  • B2B portal is custom-built exactly how you want
Cons
  • Most development effort upfront
  • You maintain all the integration code
  • Zoho API rate limits need caching strategy

New monthly cost: ~$23/mo (Vercel + DNS)

B

Zoho Commerce Storefront

Use Zoho's own e-commerce platform as the website. Native integration with Books, Inventory, CRM. Hosted by Zoho.

Pros
  • Zero integration code — everything is native
  • Fastest to launch
  • Ken manages everything from one Zoho dashboard
  • Built-in cart, checkout, order management
Cons
  • Limited design flexibility (theme-based)
  • No custom B2B portal with bespoke pricing
  • No Clerk — stuck with Zoho's auth
  • Hard to customize dealer locator, quote flows
  • Another monthly fee ($25-100/mo)

New monthly cost: ~$25-100/mo (Zoho Commerce plan)

C

Hybrid: Next.js Frontend + Zoho Commerce Backend

Use Zoho Commerce's API as a headless backend for products/orders, but build a custom Next.js frontend for full design control.

Pros
  • Design freedom of Option A
  • Commerce handles order/payment logic
  • Native Zoho ecosystem sync
Cons
  • Zoho Commerce API is limited vs Inventory/Books APIs
  • Paying for both Vercel AND Zoho Commerce
  • Two systems to manage — worst of both worlds
  • Still can't do bespoke B2B pricing natively

New monthly cost: ~$45-120/mo (Vercel + Zoho Commerce + DNS)

My Recommendation: Option A

You need a custom B2B portal with per-account pricing, a dealer locator pulling from CRM, a "Contact Dealer" email flow, and full design control. Zoho Commerce can't do most of that without heavy workarounds. Option A gives you exactly what you need, costs the least monthly, and uses the stack you already know. The extra upfront dev work pays for itself in flexibility.
